问题

1. 基础模板与事实证据被混合注入

* `build_format_data()` 将 `build_persona_injection_prompt()` 生成的基础模板注入内容与 `persona_context` 直接拼接为同一块 `combined_persona_context`，导致模型把“风格规则”当成“事实依据”使用。

2. 主流程存在两条 persona 检索链，职责冲突

* 主链路已经通过 `persona_rag_engine.recall()` 做了一次 persona 检索。
* 之后又在发送消息阶段调用 `persona_system.build_context()`，并把结果追加到 `assembled_context.slots["evidence_chunks"]`，造成证据来源重复、调试困难、fallback 行为混入主链。

3. story 检索失败后存在随机 fallback

* `PersonaContextService._build_story_context()` 在 story 检索未命中时，会从 `_story_entries()` 中随机挑选高情绪权重或任意故事作为“角色故事证据”，这会伪装成检索成功，实际并未命中用户问题。

4. persona state 持久化时丢失 story 元数据

* `_sync_new_persona_state()` 将 `persona_system.entries` 写入 `ParentChunk` 时，没有保留 `kind`、`title`、`trigger_topics`、`emotional_weight` 等 story 专属字段。
* `PersonaRAGEngine._entries()` 从 `PersonaState` 还原 entries 时，又把 metadata 中的 `kind` 固定写成 `source_chunk`，导致系统重启或读档后 story 检索能力退化。

5. state 同步时 chunk 主键不稳定

* `_sync_new_persona_state()` 中 `chunk_id` 优先使用 `entry.get("id")`，但项目里 entries 的主键实际主要使用 `chunk_id`，会造成状态同步后 chunk id 漂移。

6. story query 检索策略过宽，容易误召回

* `RAGTool.plan_query()` 默认对所有 query 开启 HyDE。
* `RAGTool.search()` 只要 `final_score > 0` 就纳入 hits，没有 story 专用最低阈值。这样会放大故事类问答的误召回。

完整修改方案

一、统一 persona 检索主链

目标

* 只保留 `PersonaRAGEngine.recall()` 作为主流程唯一 persona 检索入口。
* 删除发送消息阶段再次调用 `persona_system.build_context()` 并追加 evidence 的逻辑。

修改文件

* `main.py`

必须删除的旧逻辑

```python
local_persona_context = self.persona_system.build_context(str(user_input or ""))
if local_persona_context:
    assembled_context.slots["evidence_chunks"] = "\n\n".join(
        part for part in (assembled_context.slots.get("evidence_chunks", ""), local_persona_context) if part
    )
```



修改要求

1. persona 证据只允许来自 `_run_harness_pipeline()` 内部的 `persona_rag_engine.recall(normalized_query)` 结果。
2. `send_message()` 中不再旁路追加任何来自 `persona_system.build_context()` 的上下文。
3. `PersonaContextService` 如果保留，只能用于 preview/debug，不再参与聊天主链。

目标

* 基础模板只控制“怎么说”。
* 检索证据只控制“说什么”。


2. 返回字段改为：

3. 删除旧的混合字段 `persona_context` 的主流程用法，不再把 `style_prompt` 和 `evidence_prompt` 合并。

4. 在最终给 LLM 的模板中加入硬约束：

```text

{style_prompt}

【本轮角色事实证据】
{evidence_prompt}

【外部工具证据】
{tool_context}

【硬性规则】
1. “角色说话底色”只用于控制语气、节奏、措辞、情绪表达方式，不提供任何新事实。

   - 如果“本轮角色事实证据”中有 story_chunk 证据，可以用角色口吻转述该故事。
   - 如果没有足够 story_chunk 证据，必须明确说明资料不足，不能编造。
```

三、删除 story 检索随机 fallback

目标

* 没命中就是没命中。
* 不允许随机挑故事伪装命中。

修改文件

* `persona_context_service.py`

必须删除的旧逻辑

* `fallback_pool = self._story_entries()`
* `preferred = [...] or fallback_pool`
* `chosen_entry = random.choice(preferred)`
* 任何 `_format_story_entry(...)` 的随机 fallback 调用


修改后的 `_build_story_context()` 原则

```python
def _build_story_context(self, query: str, top_k: int = 4, char_budget: int = 1000) -> str:
    result = self.rag.search(
        query,
        self._story_entries(),
        top_k=top_k,
        query_type="story",
        filters={"kind": {"story_chunk"}},
        enable_hyde=False,
    )
    chosen = self._choose_story_hit(result.hits)
    if chosen is not None:
        return self._format_story_hit("角色故事证据", chosen, char_budget)
    return ""
```

额外要求

* `PersonaContextService` 不再接入聊天主流程，只允许作为 preview/debug 辅助器。
* 如果该类已被 `PersonaRAGEngine` 覆盖，可保留文件但移除主流程依赖。

* 添加最小阈值

```python
def plan_query(self, query: str, query_type: str = "", enable_multi_query: bool = True, enable_hyde: bool = True) -> RAGQueryPlan:
    direct_query = re.sub(r"\s+", " ", str(query or "")).strip()
    inferred_type = query_type or self._infer_query_type(direct_query)

    if inferred_type == "story":
        enable_hyde = False

    multi_queries = self._multi_query_expand(direct_query, inferred_type) if enable_multi_query else []
    hyde_document = self._hyde_expand(direct_query, inferred_type) if enable_hyde else ""
    ...
```

修改 2：search 增加最低阈值
在 `search()` 中构建 hits 时加入：

```python
min_score = 0.0
if plan.query_type == "story":
    min_score = 0.42
elif plan.query_type in {"persona", "character"}:
    min_score = 0.30

...

```

要求

1. `story` 查询至少有专用最小阈值。
2. 若是 `story` 且 filters 限定为 `story_chunk`，命中数为 0 时直接返回空 hits，不做任何补偿。

八、修正最终回答阶段的硬约束

目标

* 明确禁止把模板改写成经历
* 证据不足时必须拒绝脑补

修改文件

* 最终 prompt 模板文件
* 若模板定义在 `const.py`、`main.py` 或其他文件，统一更新

必须加入的规则

```text
[硬性规则]
1. “角色说话底色 / style_prompt”只用于控制语气、节奏、措辞、情绪表达方式，不提供任何新事实。
2. 只要回答涉及角色设定、经历、故事、过往事件，事实内容只能来自 evidence_prompt 中的证据。
3. 如果 evidence_prompt 中没有足够证据，不允许补写、脑补、润色成具体事件，不允许把基础模板改写成经历。
4. 如果 tool_context 提供了现实世界信息，则现实信息只能来自 tool_context。
5. 当用户要求“讲故事”时：
   - 如果 evidence_prompt 中存在 story_chunk 证据：可以用角色口吻转述该故事
   - 如果不存在 story_chunk 证据：必须明确说明现有资料不足，不能编造
```

九、必须删除的旧部分

1. `main.py` 中再次调用 `persona_system.build_context()` 并追加 evidence 的逻辑。
2. `persona_context_service.py` 中任何 `random.choice(...)` story fallback。
3. `ai_runtime_support.py` 中把 `injection_prompt` 和 `persona_context` 合并成单一 `combined_persona_context` 的逻辑。
4. `persona_rag_engine.py` 中把 state 还原的所有 chunk 强制写成 `source_chunk` 的逻辑。
5. story query 未命中后仍退化为普通 source chunk 故事式回答的逻辑。

十、修改完成后的验收标准

1. 用户问“讲一个她经历过的故事”

* 若命中 story_chunk：允许以角色口吻转述，但事实必须可回溯到命中的 story chunk。
* 若未命中 story_chunk：必须明确说明资料不足，不能随机讲、不能模板脑补。

2. 用户问“她的性格 / 口头禅 / 喜恶”

* 可以使用基础模板控制风格。
* 但不能把模板规则伪装成具体经历。

3. 用户问天气 / 外部知识

* 必须走 tool。
* 不走 persona RAG 伪回答。

4. 系统重启后再次问角色故事

* story 检索效果不能下降。
* 说明 `ParentChunk.kind` 与 story 元数据已成功持久化并恢复。
